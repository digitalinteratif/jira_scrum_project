KAN-122: Aggregation performance improvements (materialized views & summary tables)

Overview
--------
This doc describes the runtime objects, their refresh schedule recommendations, reindexing guidance, and testing notes for the KAN-122 changes.

Objects created by migration (Postgres only)
 - Materialized view: clicks_per_shorturl_day (columns: short_url_id integer, day date, clicks bigint)
   - Created with a UNIQUE index on (short_url_id, day) to support REFRESH MATERIALIZED VIEW CONCURRENTLY.
   - Additional index on (short_url_id, day) for read performance.
 - Summary table: analytics_summary_shorturl_daily (columns: short_url_id int, day date, clicks bigint)
   - Primary key (short_url_id, day).
   - Indexed on short_url_id.

Why both MV and summary table?
 - Materialized view provides a fast, read-optimized aggregation. Refreshing it frequently is relatively cheap for moderate datasets.
 - The summary table is useful for external reporting systems and to keep a historical snapshot that can be archived or partitioned later. Rebuilding the summary table nightly from the MV gives a stable footprint for fast totals without touching the write-heavy ClickEvent raw rows at query time.

Refresh schedule (recommended)
 - Materialized view:
    - Refresh frequency: every 15 minutes for near-real-time dashboards on high-traffic deployments; nightly is acceptable for low-traffic.
    - Use REFRESH MATERIALIZED VIEW CONCURRENTLY clicks_per_shorturl_day where possible.
 - Summary table:
    - Rebuild nightly (e.g., 02:00 UTC) from the latest MV. The nightly job truncates/upserts the entire summary table from the MV.
 - Example system cron entries:
    - Run MV refresh every 15 minutes:
      */15 * * * * cd /path/to/app && /path/to/venv/bin/python bin/refresh_analytics.py --refresh-mv-only >> /var/log/refresh_analytics.log 2>&1
    - Rebuild summary nightly at 02:00:
      0 2 * * * cd /path/to/app && /path/to/venv/bin/python bin/refresh_analytics.py --rebuild-summary-only >> /var/log/refresh_analytics.log 2>&1

Alternative: use a single job that refreshes MV then rebuilds summary (bin/refresh_analytics.py default behavior).

Reindex & maintenance guidance
 - Periodically (weekly/monthly depending on volume) run:
     VACUUM ANALYZE clicks_per_shorturl_day;
     VACUUM ANALYZE analytics_summary_shorturl_daily;
     REINDEX INDEX ux_clicks_per_shorturl_day_shorturl_day;
     REINDEX INDEX ix_clicks_per_shorturl_day_shorturl;
     REINDEX INDEX ix_analytics_summary_shorturl_daily_shorturl;
 - If REFRESH MATERIALIZED VIEW CONCURRENTLY starts failing due to missing unique index, confirm UX index exists:
     SELECT indexname FROM pg_indexes WHERE tablename='clicks_per_shorturl_day';
 - If data drift is suspected between summary table and MV, run:
     SELECT COUNT(*) FROM clicks_per_shorturl_day EXCEPT SELECT short_url_id::integer AS short_url_id, day, clicks FROM analytics_summary_shorturl_daily;
   and check for differences. Use the rebuild script to resync.

Dev fallback & testing notes
 - Migration is a no-op on SQLite. The application will run direct aggregations using ClickEvent table for development and tests.
 - The analytics endpoint contains logic to detect Postgres and prefer the summary table / MV when present; it falls back to runtime aggregations if objects are absent or on SQLite.
 - Suggested integration test:
   - Populate representative ClickEvent rows (e.g., 1M rows in a staging Postgres) and benchmark:
       a) Query time before: current analytics endpoint runtime aggregation
       b) Apply migration, refresh MV, rebuild summary
       c) Query time after: prefer MV/summary
   - Document the before/after wall-clock times and storage footprint.
 - Unit tests: add tests that verify the analytics_link_data returns correct totals/histogram given small sample datasets when using the MV vs direct aggregation.

Rollback procedure
 - Run Alembic downgrade to remove materialized view and summary table.
 - Stop any scheduled jobs (cron / systemd timers) that attempt to REFRESH the MV to prevent errors after downgrade.

Trace & logs
 - Migration and the refresh script write non-blocking traces to trace_KAN-122.txt for Architectural Memory. Inspect these for operational troubleshooting.

Contact
 - For questions about operational scheduling and indexing strategy, contact the platform operator / DB admin.

Deployment checklist (short)
------------------------------------------------------------
1) Add migration file to Alembic versions and run alembic upgrade head on Production Postgres.
2) Install bin/refresh_analytics.py on the host running scheduled jobs, make it executable.
3) Schedule refreshes (cron/systemd) per above schedule.
4) Deploy code that contains the updated analytics route (routes/analytics.py) which will automatically prefer pre-aggregated data.
5) Run the bin/refresh_analytics.py once manually to populate MV & summary.
6) Monitor trace_KAN-122.txt and analytics query response times. Optionally enable metrics/alerts if query latencies rise.
